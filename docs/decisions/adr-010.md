# ADR-010: Renderer Architecture

## Status

Accepted

---

# Context

The purpose of the ReciGraph compiler is to transform recipe knowledge into multiple presentation formats.

ReciGraph's build layer orchestrates discovery, compilation, view-model construction, rendering, and output management.

Expected outputs include:

- Static HTML website
- Cookbook PDF
- Markdown
- Shopping lists
- Search indexes
- Future presentation formats

These outputs share the same semantic information but require different presentation.

Renderers should therefore concern themselves only with presentation.

---

# Decision

Rendering is divided into two distinct phases:

```
Graph IR
      ↓
View Model
      ↓
Renderer
      ↓
Artifact
```

The compiler remains responsible for semantics.

Renderers remain responsible only for presentation.

---

# Responsibilities

## Compiler

Produces:

- Graph IR
- Compilation Trace

---

## View Model Builder

Transforms Graph IR into a presentation-specific model.

The View Model:

- contains only presentation-ready information
- contains no compiler semantics
- contains no graph traversal logic
- contains no reference resolution

---

## Renderer

Consumes a View Model.

Produces one or more output artifacts.

Renderers must not:

- resolve references
- perform substitutions
- inspect AST nodes
- inspect compiler state
- depend on compiler-specific orchestration objects
- traverse the Graph IR

---

# Renderer Protocol

Every renderer implements the same interface.

```python
class Renderer(Protocol):
    name: str

    def render(
        self,
        model: ViewModel,
        output_dir: Path,
    ) -> RenderResult: ...
```

---

# RenderResult

A renderer returns information describing what it produced.

Example:

```python
RenderResult(
    artifacts=[
        "index.html",
        "recipes/vanilla.html",
    ]
)
```

The compiler does not interpret renderer output.

---

# View Models

A View Model is specific to an output family.

Examples:

RecipeViewModel

ShoppingListViewModel

SearchIndexViewModel

Each renderer family owns its own View Model.

View Models are immutable.

---

# Renderer Independence

Renderers must be deterministic.

Given identical View Models they produce identical output.

Renderers may use:

- Jinja templates
- Markdown templates
- HTML templates

or equivalent technologies.

---

# HTML Renderer (v1)

The first renderer is intentionally minimal.

It produces:

```
dist/

    index.html

    recipes/

        vanilla.html
```

No client-side JavaScript is required.

No search functionality is required.

No styling beyond basic CSS is required.

The objective is to validate the compiler pipeline.

---

# Template Logic

Templates should contain only presentation logic.

Templates should not:

- calculate ingredient substitutions
- resolve references
- inspect Graph IR
- infer semantic relationships

If template complexity grows, the required information should instead be added to the View Model.

---

# Extensibility

Future renderers should require only:

- a View Model builder
- a Renderer implementation

The compiler remains unchanged.

---

# Consequences

## Positive

- Clear separation of concerns.
- Thin renderers.
- Shared semantic interpretation.
- Easier testing.
- Renderer independence.

## Negative

- Introduces an additional transformation stage.
- Requires View Model definitions.

This cost is considered acceptable because it isolates presentation concerns from compiler concerns.

---

# Design Principle

The compiler understands recipes.

The renderer understands presentation.

The View Model is the contract between them.

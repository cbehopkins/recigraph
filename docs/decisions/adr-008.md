# ADR-008: References Use a Stable Dotted Identifier Syntax

**Status:** Accepted

## Context

Relationships between entities are fundamental to the ReciGraph DSL.

Examples include:

* A step targeting an ingredient.
* A procedure including another procedure.
* A procedure extending another procedure.
* Future references to equipment, containers, mixtures and other domain entities.

The reference syntax should be:

* Easy for humans to author.
* Easy to review in version control.
* Stable over time.
* Independent of the compiler's internal implementation.

The compiler itself is expected to evolve as the language matures, and internal representations should not constrain the authoring experience.

## Decision

All entity references in the ReciGraph DSL use the following syntax:

```text
<entity_type>.<identifier>
```

Examples:

```text
ingredient.whole_milk
ingredient.double_cream
procedure.classic_vanilla_base
procedure.coffee_base
equipment.blender
container.creami_pint
```

References are represented as plain strings within the authored YAML.

For example:

```yaml
target: ingredient.whole_milk
```

The authoring format is considered part of the language specification and should remain stable.

The compiler is free to parse these strings into richer internal representations if future requirements justify doing so, but this must remain an implementation detail.

## Rationale

This syntax provides several advantages:

* Compact and easy to author.
* Readable in code reviews.
* Easy to search using standard text tools.
* Simple to document.
* Consistent across all entity types.
* Independent of any particular implementation language.

The syntax also follows the KISS principle by avoiding unnecessary verbosity in the DSL.

## Consequences

The DSL remains concise and approachable.

Compiler phases may initially work directly with reference strings.

Future compiler implementations may introduce richer internal reference objects without requiring any changes to authored YAML.

The parser, resolver and validator are responsible for interpreting references; authors are not exposed to the compiler's internal representation.

## Alternatives Considered

### Structured YAML Objects

For example:

```yaml
target:
  type: ingredient
  id: whole_milk
```

This was rejected because it significantly increases the verbosity of the DSL without providing meaningful benefits for authors.

### Immediate Object Resolution

The loader would replace references with direct object instances during parsing.

This was rejected because it couples parsing to semantic analysis, introduces ordering dependencies and blurs the responsibilities of compiler phases.

### Typed Internal Reference Objects

The compiler may choose to represent references internally using dedicated types such as:

```python
EntityReference(
    entity_type="ingredient",
    entity_id="whole_milk",
)
```

This remains a valid implementation strategy, but it is explicitly an implementation detail and must not affect the external DSL.

## Notes

This decision deliberately separates the language specification from the compiler implementation.

The ReciGraph DSL defines **how authors express references**.

The compiler is free to optimise **how those references are represented internally**.

Maintaining this separation allows the implementation to evolve without breaking authored content or changing the language itself.

# ADR-007: Domain First, Presentation Second

**Status:** Accepted

## Context

ReciGraph is intended to generate multiple outputs from a single source of truth.

Current planned outputs include:

* Static HTML websites
* Printable PDF cookbooks
* Shopping lists
* Ingredient indexes
* Search indexes
* AI-ready knowledge bases
* Future mobile applications and APIs

Each of these outputs has different presentation requirements.

A common failure mode for projects of this nature is allowing the needs of one renderer to influence the underlying data model. Over time, presentation-specific fields accumulate in the core domain model, making it harder to support new outputs and reducing the reusability of the underlying knowledge.

## Decision

The ReciGraph DSL models the **culinary domain**, not any specific presentation.

Entities should represent real concepts within the recipe domain, such as:

* Ingredients
* Ingredient Families
* Actions
* Stages
* Procedures
* Equipment
* Containers
* Relationships between those concepts

The DSL must not contain fields whose sole purpose is to support a particular renderer or output format.

Presentation-specific concerns belong in renderers or view models, not in the domain model.

The compiler pipeline will therefore maintain a clear separation:

```text
Authoring DSL
        │
        ▼
Domain Model
        │
        ▼
Resolved Model
        │
        ▼
View Model
        │
        ▼
Renderer
```

Renderers are responsible for deciding how domain concepts are presented.

The domain model is responsible only for representing those concepts accurately.

## Consequences

The same knowledge base can generate multiple outputs without modifying the authored data.

New renderers can be introduced without changing the DSL.

The domain model remains focused, consistent and easier to evolve.

Presentation logic remains isolated from compiler logic.

Future features should be evaluated by asking:

> "Does this represent a genuine domain concept, or is it only needed by a particular renderer?"

If the answer is "only a renderer", the feature should not be added to the DSL.

## Examples

### Good

A procedure records that it includes another procedure.

This is a domain relationship that exists regardless of whether the recipe is rendered as HTML, PDF or another format.

### Bad

Adding a field such as:

```yaml
heading_level: 2
```

to control HTML output.

Heading levels are a presentation concern and belong entirely within the renderer.

### Good

A procedure records an estimated preparation time.

Preparation time is an intrinsic property of the recipe and may be displayed by any renderer.

### Bad

Adding a field to specify whether preparation time should appear in bold on the website.

Text styling is a presentation concern and must not influence the DSL.

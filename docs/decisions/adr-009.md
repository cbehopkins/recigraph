# ADR-009: Compiler Intermediate Representation (Graph IR)

## Status

Accepted

---

# Context

The ReciGraph compiler transforms recipe definitions written in the ReciGraph DSL into multiple downstream outputs, including:

- Static website
- Cookbook PDF
- Shopping lists
- Search index
- AI tooling
- Future renderers

Initially it may appear sufficient for renderers to consume the parsed DSL (AST) directly.

However, doing so couples renderers to the source language and forces each renderer to independently perform reference resolution, substitution handling and graph analysis.

This duplicates compiler logic, increases implementation complexity and makes future evolution more difficult.

---

# Decision

The compiler SHALL produce an explicit **Intermediate Representation (IR)**.

This IR is a fully resolved, materialised graph representing the semantic meaning of a recipe after compilation.

Renderers consume the IR rather than the AST.

The AST remains an internal representation of the authored DSL.

---

# Compilation Pipeline

The compiler architecture becomes:

```
YAML

    ↓

AST (Pydantic Models)

    ↓

Validation

    ↓

Reference Resolution

    ↓

Graph Transformation

    ↓

Graph IR

    ↓

Renderers
```

The Graph IR forms the stable boundary between compilation and rendering.

---

# Responsibilities

## AST

Represents:

- authored YAML
- syntactic structure
- references
- source locations
- validation targets

The AST is a faithful representation of source documents.

It intentionally preserves author intent rather than compiler interpretation.

---

## Graph IR

Represents:

- resolved entities
- resolved references
- graph relationships
- derived entities
- substitutions
- transformation provenance
- compiler metadata

The IR represents semantic meaning rather than authored structure.

---

# GraphState

The existing `GraphState` model is considered the initial implementation of the Graph IR.

Over time it is expected to evolve into a richer representation.

This evolution is considered an internal compiler concern.

The public DSL is unaffected.

---

# Renderers

Renderers MUST consume the Graph IR.

Renderers SHOULD NOT:

- resolve references
- perform substitutions
- analyse dependency graphs
- interpret compiler semantics

These responsibilities belong exclusively to the compiler.

---

# Compiler Responsibilities

The compiler is responsible for producing a fully resolved IR.

This includes:

- reference resolution
- validation
- graph transformation
- application of Step-local bindings
- generation of derived entities
- provenance tracking

---

# Trace

Compilation produces two outputs:

- Graph IR
- Compilation Trace

The trace records each graph transformation performed during compilation.

It is intended for:

- debugging
- testing
- diagnostics
- future visualisation

The trace is not part of the Graph IR itself.

---

# Design Principles

The Graph IR should:

- be deterministic
- be immutable
- be renderer-independent
- preserve semantic meaning
- hide compiler implementation details
- evolve independently of the DSL

---

# Consequences

## Positive

- Clear separation between compilation and rendering.
- Single source of truth for recipe semantics.
- Simpler renderers.
- Easier testing.
- Better support for future tooling.
- Enables provenance tracking.
- Enables graph visualisation.
- Enables future optimisation passes without changing renderers.

## Negative

- Introduces an additional compiler representation.
- Requires maintenance of AST → IR transformation.

These costs are considered acceptable given the reduction in duplicated logic and the improved architectural separation.

---

# Future Evolution

The Graph IR is expected to grow over time to include richer semantic information, such as:

- dependency relationships
- ingredient provenance
- nutritional metadata
- equipment usage
- timing information
- optimisation metadata

Such evolution should not require changes to the public DSL.

---

# Alternatives Considered

## Render directly from the AST

Rejected.

This would require every renderer to duplicate compiler logic and would tightly couple rendering to the authored DSL.

---

## Generate renderer-specific models directly

Rejected.

This would make the compiler responsible for presentation concerns and prevent reuse of semantic information across renderers.

---

## Make the Graph IR part of the public DSL

Rejected.

The IR is an internal compiler representation.

Only the DSL is part of the public contract.

---

# Rationale

ReciGraph is fundamentally a compiler rather than a document conversion tool.

The authored YAML describes recipes.

The compiler derives their semantic meaning.

The Graph IR captures that meaning in a renderer-independent form, providing a stable architectural boundary between the language frontend and all downstream consumers.
